Guía simple del proyecto

Arquitectura Harness para Codex

Este documento explica, en palabras simples, para qué sirve cada carpeta y archivo del Harness, cómo se conectan entre sí y cómo usarlo como plantilla para proyectos nuevos.

Requisitos antes de código Arquitectura documentada Seguridad revisada Pruebas planeadas PRs pequeños Agents + skills

¿Qué es este Harness?

Es una estructura de trabajo para que Codex no salte directo a programar. Funciona como una lista de control inteligente: primero se entiende el problema, luego se escriben requisitos, se diseña la arquitectura, se revisa seguridad, se planean pruebas y recién después se implementa.

En corto: el Harness convierte una idea suelta en un plan de entrega mantenible, revisable y seguro.

Explicado fácil Imagina que quieres construir una casa. Este Harness evita empezar comprando ladrillos sin plano. Primero define qué casa quieres, quién la usará, qué riesgos hay, cómo se probará y cómo se entregará por partes.

El flujo completo

Este es el recorrido ideal de una idea hasta llegar a código revisado.

1

Idea

Alguien pide una función, arreglo o mejora.

2

Spec

Se documentan objetivo, usuarios, requisitos y criterios de aceptación.

3

Diseño

Se define arquitectura, límites, componentes, datos e integraciones.

4

Seguridad

Se revisan auth, permisos, secretos, datos y riesgos de IA si aplica.

5

Pruebas

Se planean unitarias, integración, regresión y evaluación de IA.

6

Plan

Se decide cómo implementar y cómo dividir el cambio en PRs pequeños.

7

Código

Codex implementa siguiendo la spec, diseño, ADRs y pruebas.

8

Review

Se revisa calidad, seguridad, pruebas, rollback y documentación.

9

Entrega

Se valida Definition of Done y se deja evidencia.

10

Aprendizaje

Los ADRs y reviews quedan como memoria para el próximo cambio.

La idea clave

.codex le dice a Codex cómo debe trabajar. docs guarda la evidencia del trabajo. src y tests son donde eventualmente vivirán el código y las pruebas reales.

Si solo tienes instrucciones pero no documentos, no hay trazabilidad. Si solo tienes documentos pero no workflows, Codex puede trabajar de forma inconsistente. Este Harness junta las dos cosas.

Diferencia entre agent y skill

Esta es una de las ideas más importantes del Harness. Un agent y una skill no son lo mismo.

Agent

Es el especialista

Un agent le dice a Codex con qué mentalidad debe pensar. Por ejemplo: "piensa como arquitecto", "piensa como revisor de seguridad" o "piensa como tester".

Ejemplo fácil: si estás construyendo una casa, el agent sería la persona arquitecta que entiende planos, espacios, riesgos y estructura.

Skill

Es la receta

Una skill le dice a Codex qué pasos debe seguir para hacer una tarea. Por ejemplo: revisar arquitectura, escribir una spec, planear pruebas o dividir PRs.

Ejemplo fácil: si estás cocinando, la skill sería la receta paso a paso para preparar bien el plato.

Juntos

Especialista + receta

El agent pone el criterio. La skill pone el método. Juntos hacen que Codex piense bien y trabaje ordenado.

Ejemplo fácil: "Chef" es el agent. "Receta de pastel" es la skill. El chef usa la receta para no improvisar.

Pregunta Agent Skill
¿Qué es? Un rol o punto de vista. Un proceso o checklist.
¿Para qué sirve? Para pensar como especialista. Para seguir pasos repetibles.
Ejemplo del repo .codex/agents/architect.md skills/architecture-review/SKILL.md
En palabras simples "Quién piensa" "Qué pasos sigue"

Cómo se comunican agents y skills

No se conectan como código con un import. Se conectan por instrucciones escritas en los workflows y en los propios agents.

1

Workflow

Dice qué rol usar y qué skill aplicar.

2

Agent

Codex adopta la mentalidad del especialista.

3

Skill

Codex sigue el checklist o proceso.

4

Template

Codex usa un formato para escribir el resultado.

5

Docs

El resultado queda guardado como evidencia.

Ejemplo con Architect

En .codex/workflows/feature-delivery.md el flujo dice:

3. Use Architect 4. Apply Architecture Review skill

Eso significa: primero Codex piensa como Architect, usando las reglas de .codex/agents/architect.md. Luego aplica la receta de skills/architecture-review/SKILL.md para revisar límites, componentes, integraciones, riesgos, tradeoffs y ADRs.

Product Analyst + SDD

El agent entiende el problema de negocio. La skill sdd convierte eso en una especificación.

Architect + Architecture Review

El agent diseña la solución. La skill architecture-review obliga a revisar riesgos, límites, componentes y ADRs.

Security Reviewer + Security Review

El agent piensa en amenazas. La skill security-review guía la revisión de auth, permisos, secretos y datos.

Test Engineer + TDD

El agent piensa en calidad. La skill tdd define pruebas antes o junto con la implementación.

Code Reviewer + PR Chain

El agent revisa mantenibilidad. La skill pr-chain ayuda a dividir cambios grandes en PRs pequeños.

AI Foundry Engineer + AI Foundry

El agent piensa en IA empresarial. La skill ai-foundry guía agentes, RAG, evaluación y observabilidad.

La frase para recordarlo El agent decide "con qué sombrero piensa Codex". La skill decide "qué pasos sigue Codex".

Cómo se conectan las carpetas

1. La regla manda

AGENTS.md establece que Codex debe hacer discovery, spec, diseño, seguridad, pruebas, plan de implementación y review antes de entregar cambios importantes.

2. El workflow guía

.codex/workflows/feature-delivery.md dice el orden específico para una nueva feature y qué documentos deben existir.

3. La plantilla da forma

.codex/templates evita documentos improvisados. Cada artefacto tiene campos de estado, riesgos, evidencia y aprobación.

4. Docs guarda prueba

docs guarda lo que se aprobó. Después el código en src debe respetar esa evidencia.

Los roles especialistas

Estos roles no son personas reales dentro del repo. Son instrucciones para que Codex piense con el sombrero correcto.

Product Analyst

Convierte una idea en requisitos, usuarios, viajes, reglas de negocio y criterios de aceptación.

Architect

Define límites, componentes, integraciones, despliegue, escalabilidad, observabilidad y ADRs.

Security Reviewer

Busca riesgos de autenticación, autorización, secretos, fuga de datos, tenant isolation y seguridad de IA.

Test Engineer

Define pruebas, cobertura, regresión, quality gates y evaluaciones para sistemas con IA.

DevOps Engineer

Piensa en CI/CD, infraestructura, ambientes, monitoreo, alertas, rollback y producción.

Code Reviewer

Revisa que el código sea correcto, simple, mantenible, testeable, seguro y fácil de revertir.

AI Foundry Engineer

Se usa cuando el proyecto involucra agentes, RAG, Azure OpenAI, AI Search o Microsoft AI Foundry.

Cómo usarlo en un proyecto real

Para una feature nueva

Abre el workflow .codex/workflows/feature-delivery.md. Luego crea una spec, diseño, revisión de seguridad si aplica, test plan, implementation plan y PR plan.

Solo después de eso empieza el código en src.

Para un bug

Usa .codex/workflows/bug-fix.md. Primero reproduce el bug, define comportamiento esperado, encuentra causa raíz, crea una prueba de regresión y arregla lo mínimo.

Para una decisión técnica

Crea un ADR en docs/adr. No dejes decisiones grandes escondidas en conversaciones. El futuro equipo necesita saber por qué se eligió una ruta.

Para reutilizarlo

Sube este repo a GitHub como template. En cada proyecto nuevo, copias esta estructura o creas el repo desde la plantilla.

Ejemplo mental rápido

Supón que quieres crear un bot para proveedores. Sin Harness, podrías decir: "Codex, hazme el bot". Eso normalmente produce código rápido pero con huecos.

Con Harness, el proceso sería:

  1. Definir quién usa el bot y para qué.
  2. Documentar qué puede y qué no puede hacer.
  3. Diseñar si usará API, base de datos, ERP, RAG o agente de IA.
  4. Revisar permisos, datos sensibles, secretos y auditoría.
  5. Crear pruebas para respuestas, tool calling, seguridad y regresión.
  6. Dividir la implementación en PRs pequeños.
  7. Implementar, probar, revisar y dejar evidencia.

Ese es el valor: Codex se vuelve más disciplinado y el proyecto queda más fácil de mantener.

Reglas simples para no perderse

Si no está escrito, no existe

Las decisiones importantes deben quedar en docs, no solo en chat.

Primero el porqué, luego el código

Antes de implementar, entiende objetivo, usuario, riesgos y criterios de aceptación.

Seguridad no es prompt

Los permisos se controlan en arquitectura, API, datos y herramientas, no solo con instrucciones al modelo.

PR pequeño gana

Menos líneas por PR significa review más rápido, rollback más simple y menos sorpresas.

Todo debe poder probarse

Cada requisito importante debe tener prueba, evidencia o validación manual explícita.

ADR para decisiones grandes

Si una decisión afecta el futuro del sistema, crea un ADR.
