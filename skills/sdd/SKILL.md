# Spec Driven Development (SDD)

Purpose

Convert ideas and requirements into implementation-ready specifications before coding.

Never start implementation before completing this process.

---

# Workflow

1. Discovery
2. Requirements Analysis
3. Business Rules
4. Functional Requirements
5. Non-Functional Requirements
6. Acceptance Criteria
7. Risks
8. Assumptions
9. Open Questions
10. MVP Scope

---

# Discovery

Identify:

- Problem
- Users
- Goals
- Constraints
- Success Criteria

Ask for clarification when information is missing.

Do not assume business requirements.

---

# Requirements

Document:

- Functional Requirements
- Non-Functional Requirements
- Business Rules
- User Journeys

Separate mandatory requirements from future enhancements.

---

# Acceptance Criteria

Every feature must have acceptance criteria.

Format:

Given
When
Then

---

# Deliverables

Generate:

/docs/specs/[feature-name].md

Include:

- Executive Summary
- Goals
- Personas
- User Journeys
- Requirements
- Acceptance Criteria
- Risks
- MVP Scope

---

# Exit Criteria

Implementation may start only when:

- Requirements are clear
- Acceptance criteria exist
- Risks are documented
- Open questions are resolved or tracked
